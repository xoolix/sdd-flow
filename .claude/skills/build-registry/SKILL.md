# Build Skill Registry

Scan all project-specific skills in `.claude/skills/`, generate **compact rules** (5-15 lines per skill), and write the registry to `.claude/skills/skill-registry.md`.

> The registry is what orchestrators (`sdd-next`, `sdd-auto`) inject into sub-agents instead of full skill files. This saves tokens: ~100 lines of compact rules vs ~2000 lines of full skills.

## Steps

1. **Scan for project skills** â€” List all directories in `.claude/skills/`. Skip core SDD skills:
   `new-feature`, `plan-feature`, `implement-task`, `simplify-code`, `review-feature`, `archive-feature`, `research-spike`, `sdd-new`, `sdd-next`, `sdd-auto`, `init-project`, `build-registry`, `architecture-map`, `_shared`

2. **For each project skill found**, read its `SKILL.md` and extract:
   - `name` from frontmatter
   - `description` from frontmatter
   - `applies-to` from frontmatter (if present)
   - The actionable rules from the skill content

3. **Generate compact rules** for each skill â€” distill the full skill into **5-15 lines** of actionable constraints:
   - Include ONLY rules that a sub-agent must follow when writing/reviewing code
   - NO motivation, NO explanation of why, NO examples longer than 1 line
   - NO skill purpose or description (that's in the trigger table)
   - Format as a bullet list of imperative rules
   - If the skill has code examples, extract the pattern as a one-line rule (not the full example)

4. **Determine phase mapping** for each skill using this priority:
   - If `applies-to` exists in frontmatter â†’ use it
   - If skill is listed in `.claude/skills/skill-map.md` â†’ use that mapping
   - Otherwise â†’ default to `implement-task, review-feature`

5. **Write the registry** to `.claude/skills/skill-registry.md`:

```markdown
# Skill Registry
<!-- Auto-generated by /build-registry. Do not edit manually. -->
<!-- Regenerate with: /build-registry -->

## Trigger Table

| Skill | Description | Phases |
|-------|------------|--------|
| react-19 | React 19 patterns and conventions | implement-task, review-feature |
| playwright-e2e | E2E testing with Playwright | implement-task |

## Compact Rules

### react-19
- Use `use()` hook for promises, not useEffect+useState
- Prefer Server Components by default, add `'use client'` only when needed
- Use `useActionState` for form submissions, not onSubmit handlers
- Use `useOptimistic` for optimistic UI updates
- Wrap lazy components in `<Suspense>` with meaningful fallbacks

### playwright-e2e
- Use `page.getByRole()` and `page.getByText()` over CSS selectors
- Each test must be independent â€” no shared state between tests
- Use `test.describe` for grouping, `test.beforeEach` for setup
- Assert with `expect(locator).toBeVisible()` before interacting
```

6. **Present summary** â€” Show how many skills were processed and the registry location.

## Result envelope

```
## Result
- **Status**: success | partial | blocked
- **Summary**: [N skills scanned, registry generated at .claude/skills/skill-registry.md]
- **Artifacts**: [.claude/skills/skill-registry.md]

- **Risks**: [skills without clear patterns may have weak compact rules â€” review manually]
```

## Rules
- **NEVER use Plan Mode**
- Compact rules must be **actionable constraints**, not documentation
- Each rule should be something a code reviewer could check: "did the code follow this rule?"
- If a skill is too vague to extract rules from, include a single rule: `- Follow {skill-name} patterns from .claude/skills/{name}/SKILL.md`
- Deduplicate: if two skills have the same rule, keep it in the more specific skill
- Project-level skills override user-level skills with the same name
- Always output the result envelope at the end
